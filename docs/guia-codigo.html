◀ el diagnóstico y el árbol de deducción vuelven por el mismo camino

Regla de oro: la inteligencia vive 100% en el .clp. El backend no razona —solo traduce datos— y el front no razona —solo presenta—. Si un diagnóstico te parece raro, el origen está en las reglas, no en Python ni en React.
01

El recorrido de una petición

  1. El front arma un objeto con las 3 capas (contenedores, nodos, tráfico, red) y lo envía como JSON a POST /diagnosticar.
  2. El backend valida la forma del JSON con Pydantic y se lo pasa a motor.diagnosticar(...).
  3. motor.py abre un entorno CLIPS nuevo, convierte el JSON en hechos (assert …) y dispara CORRELACION → MITIGACION.
  4. El motor deja hechos diagnostico y justificacion; el backend los lee y los devuelve como JSON.
  5. Al pulsar ¿Por qué?, el front pide GET /por-que y muestra el árbol de deducción que dejó cada regla.
02

Backend

La capa que habla con CLIPS

backend/motor.py
  • _entorno() — crea un entorno nuevo por petición (el motor es stateful: no arrastra hechos entre diagnósticos).
  • construir_asserts() — traduce el JSON a strings (assert …); aquí pasa cpu_pct → cpu-pct.
  • diagnosticar() — orquesta: asserts → focus → run → leer resultados.
  • ErrorTelemetria — si un dato viola un rango/dominio, se convierte en HTTP 422.

La API HTTP

backend/main.py
  • Modelos Pydantic — definen el contrato del JSON; primera barrera de validación (forma y tipos).
  • CORS abierto — el front corre en otro origen y necesita permiso.
  • _ultimo — cachea el último resultado para servir /por-que sin re-ejecutar el motor.

06

Las 3 heurísticas

cascada-oomcrítica

Se dispara: contenedor OOMKilled + tasa 5xx ≥ 5%.

Causa raíz: límite de memoria insuficiente. Los 5xx y la latencia parecen un problema de red, pero son efecto del reinicio cíclico.

cpu-falsa-alarmaalta

Se dispara: CPU ≥ 90% + nodo con DiskPressure/MemoryPressure.

Causa raíz: presión de recursos del nodo. Escalar el pod empeora la situación: el cuello de botella es el nodo.

fuga-conexionesalta

Se dispara: conexiones ≥ 1000 + CPU < 90% + estado de crash.

Causa raíz: bug de código (leak). Escalar o subir límites solo retrasa el colapso; no es falta de capacidad.

Cuando ninguna aplica, la regla sin-diagnostico (salience baja) evita que el sistema quede mudo.
